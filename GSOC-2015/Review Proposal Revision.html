Detailed Project Description :

After going through PGD's code base, existing issues in the issue tracker page, tinkering with the site by running it from my localhost, having a long chat with my mentor(Mathuin) discussing the the project idea in detail, understanding the significance of various issues in the issue tracker page, feasibility of working on them, and from my observations on the features that this project lacks currently, I propose to work on the following list of features/extensions/improvements for PGD during this summer. 

  1. Phase 1 : Testing the current development version of Django(1.6.5) and it merging into master branch.

    1. Over the last summer the Django version of PGD was upgraded to 1.6.5, which is currently under the development branch and requires to be tested before merging with the master branch for release.
    2. This functional testing will be done across various components of django in the code base with respect to the changes made in Django between the versions, by writing code to automate the process of testing the existing code. 
    3. The testing would mainly be focussed on PGD code pertaining to the django changes over “models”, “views”,”templates”, “url”, “forms”. The positive outcome of tests would be our desired results, being consistent across the different versions of Django. 
    4. This will be the first objective in this project as it would provide me an in-depth insight into the PGD codebase. Since this is the first task, I am planning to start this a week or so before the coding phase begins so that I will be having a head start in this project
    5. Estimated time : 1 week.


  2. Phase 2 : Building Occupancy awareness into PGD
    1. Expanding the database by adding two more columns to the “residues” table.
      1. These fields would be storing OCCm and OCCs values as single precision floating point number.
      2. After adding the fields, few basic tests will be performed to check if the values are being saved to the database.
    2. Dumping the new database columns(OCCm and OCCs)
      1. The new column will be dumped in between the “Bg” and “H” bond columns in the statistics data dump tab-separated value file.
    3. Adding OCCm and OCCs to the search page
      1. The search page will allow the user to specify a filter for OCCs and OCCm search.
      2. Occupancy will be treated like Mobility on the search page which uses multiple rows, one each for different types of mobility.
    4. Estimated time : 1 week


  3. Phase 3 : Revamping the current user account system in PGD :

    Currently the User system provided by Django is being used only to save the searches being made by the user(currently broke). The following changes will be made as a part of revamping the User account system.

    1. Adding and/or fixing the preliminary (missing) features
      1. A new interface (or page), will be created from where a user can change his account specific details like change password, change email-id etc. This will be made accessible to all the logged in users at <web site>/account/<user id>/settings.
      2. A new button will be added in the login page that will allow the users to change the password by generating a one time use link which would be mailed to the user's registered email id. 
      3. The users will be allowed to save searches to their account, modify the saved searches and save it back, delete the saved searches. (existing feature that does not work). This feature is already implemented and is broke currently. I will be testing the existing implementation as uncertainty prevails whether these are partially implemented functionalities. As a part of TDD, this will be functional testing, where I will be writing code to test whether the implemented functionalities work as expected.(Estimated time : 1 day)
      4. A bug that I found in the current user account system is that, if the user fails to activate his/her account within the timespan of 24hrs, that username could not be used anymore. I will be solving this issue before the coding phase.
        Link : https://code.osuosl.org/issues/18069 
      5. Estimated time : 1 week 

    2. Profile page for Users
       
      1. The main purpose of this profile page is to list all the searches that the user has saved and has kept the visibility public. 
      2. The link to the profile page of the users would be like
        /accounts/<unique user id>
      3. The profile page of the users will be having relevant information about the user that others could find informative. These would be, the user's country, time-zone, area of research, age, a display picture(maybe), etc..(The additional informations will be added on the go). The details will be gathered from the user at the time of registration or post email verification (To be confirmed). 
      4. This objective would involve creating a custom User Model for saving the relevant data, which basically will be a wrapper over the Django’s default "User" model, having additional appropriate data specific to individual users.
      5. The estimated time : 1 week 
      6. Functional tests will be written to check if the implementation is working as expected. (Estimated time : 2 days)

    3. Search tab for searching users.
       
      1. Now that a profile page for users has been created having the saved search results that the user intend to share publicly, next-up will be how a user will navigate to other user's profile. The first instinctive action would be to search a user from a search tab. A new search tab will be created which will ease the process of navigation across the user profiles.
      2. This search tab will search our custom User model in the database, will match by user's first name or user's last name, retrieves the link to that User's profile and returns it.
      3. I will be using AngularJS framework in the frontend for autocompletion of user name suggestions. 
      4. If a case arises, when a particular user name is searched and multiple users exists either with the same first name or last name, all the matched search results will be displayed with names in alphabetical.
      5. Estimated time : 1 week
      6. Functional tests will be written to check if the implementation is working as expected. (Estimated time : 2 days)

    4. Searching among saved searches

      1. Since it will be a tedious process for any user to search among other user’s profile one by one for a desired saved search results, a new search bar will be created from where the saved searches can be searched.
      2. All the searches that are saved will henceforth be having an unique search id(The primary key in the new table).
      3. The searches will allow the user to add tags to the search before saving it. This tag(s) will be of string type and would be stored as “list” type object in the database. 
      4. Steps will of course be taken to protect the privately saved search results.
      5. Estimated time : 1 week
      6. Functional tests will be written to check if the implementation is working as expected. (Estimated time : 2 days)

    5. Estimated time for this phase: 4 weeks. I have reserved 3 more days after this phase, just in-case if something goes wrong and fixing that takes more than allocated time  


  4. Phase 4 : Implementing search by deposition date:

    1. The pattern of the date of deposition in the protein file will be observed across all the files(This would be done before the coding phase starts).
    2. The deposition dates in the protein file will be retrieved by parsing through the file matching the deposition date by regular expression from "pgd_splicer". 
    3. A new field will be created and the type  (a drop down of the dates, a date selection widget, or a simple text field that would take the date as seconds since the epoch on midnight of the date submission, etc) hasn’t been decided yet and would be finalised after talking with the PIs and understanding their needs and interests.
    4. The models will be modified to accept the new data and this will be tested from the Django console.
    5. Estimated time : 1 week.


  5. Phase 5 : Upgrading Django version to 1.8

    1. The current code PGD in the development branch(which is compliant with Django 1.6.5) will be upgraded to Django 1.8
    2. For this upgradation to the new version, all the backwards-incompatible changes and deprecated features in the current codebase will be removed or changed in accordance with the implementation requirement of the newer version.
    3. This task will be mainly based on code refactoring and re-architecture of the existing code base.
    4. Estimated time : 2 weeks


  6. Phase 6 : Bugfixing and Unit testing

    1. The code written post midterm will be tested by writing unit tests.
    2. The code in the repository will be hosted on the official PGD website and a real world testing will be done.
    3. Any possible bugs is generated would be fixed.
    4. A detailed documentation of the work done during the summer will be made
    5. Estimated time : 2 weeks


Stretch Goals: 

If I am able to complete the objectives before the estimated time, I will work on:

    1. Saving the downloaded search files(TSV) back to the database.
    2. Extending support for large issues.
    3. Implementing Django’s Caching framework for enhancing the performance of the searches.

Note :

  • The order of execution of the objectives in this proposal has been planned keeping in mind the requirement(s) of the successive tasks.
  • The execution of all the objectives/milestones described above will be done by TDD(Test Driven Development).
  • Currently I have proposed to do re-architecturing of PGD, making it compliant with Django1.8. If the work on "Support for large issues" is ready to be started by the time I reach that milestone, I will go ahead working on "Support for large issue" , adding the re-architecture to Django 1.8 into Stretch goals
